While implementing this application, I encountered two notable challenges that required debugging and experimentation. The first issue arose during the initial setup of the login and logging systems (commit e162894). At that stage, the login controller was querying the Users table and comparing passwords directly, but since bcrypt had not yet been integrated, the password comparison logic needed to be restructured once hashing was added in a later commit. This taught me the importance of planning the full authentication flow — including hashed storage and comparison — before wiring up the login route, rather than adding security incrementally. The second issue was a corruption of the login controller that occurred after a merge and required a dedicated fix in commit fdc3879. The first attempt at the signup route (commit 92f34e3) added input validation and trimming correctly, but a subsequent merge introduced garbled code into login.js — comment text accidentally ended up embedded inside code blocks (for example, the opening brace of the signup handler contained the comment text, and the else branch had a fragment of the success message string mixed into the logic), and the logout route was also broken as a result. Commit fdc3879 fixed all of these issues by restoring the correct structure of the signup handler and logout route. This experience highlighted how merges can introduce subtle corruption that is not immediately obvious, and reinforced the habit of reviewing the full diff after every merge before continuing development.
